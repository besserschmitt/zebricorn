The Strategic Context

The Strategic "Why": Real-World Group Constraints

The Operational Friction

Traditional athletic program sheets scale poorly in volunteer community environments. Coaches face three core variables during every session:

  • Logistical Bottlenecks: Limited mobile fitness gear creates station congestion.
  • Dynamic Attendance: Roster fluctuations break fixed group pairings.
  • Biomechanical Stress: Diverse participant age groups require programmatic safety scalings on demand.

The Core Objective

Design a zero-maintenance, highly reactive platform that computes optimized programming in seconds directly on an instructor's mobile device.

  • High Velocity: Minimize time from context inputs to visual workout map.
  • Extreme Resilience: Prevent crashes caused by unstable network environments.
  • Zero Maintenance Cost: Leverage standard open-source free tiers.

The Tactical "How": Separation of System Tiers

1. Immutable States

Decouple raw inventory, geographic constraints, and user entities from application execution layers. Rely on strict foreign key references and constraints.

2. Context-Driven Logic

Isolate business rules into clean, functional code vectors. Pass weather telemetry and structural configuration levels to pure python generation loops.

3. Re-Render Isolation

Contain and shield reactive UI layers. Ensure changes to member interaction contexts (like RSVP clicks) never trigger costly full-stack re-computations.

Phase 1: Local Genesis

The Prototyping mapping mix: Lessons Learned

The Rapid Development Footprint

The initial local archive workspace was a collection of single-purpose logic files developed during a high-focus period:

  • build_exercise_hub.py / seed_exercises.py: The initial definitions for available body vectors.
  • build_weather_logic.py: Experimental rule setups mapping basic weather states.
  • hardcore_xp_engine.py: Core gamification components used to map and track member motivation metrics.

Architectural Limitations

While the local engine was functional, it hit scaling limits due to its unified scripting pattern:

  • Tight Coupling: No separation between logic blocks and file storage paths.
  • Concurrency Limitations: SQLite file locks blocked shared co-instructor scheduling workflows.
  • Field Instability: Executing scripts inside a terminal window is unusable for coaches active on field terrains.

v1.x: Moving into the Cloud Environment

The Infrastructure Jump

To support multiple coaches and live access, the application was decoupled from local computing hardware in mid-April 2026:

  • Supabase Onboarding: PostgreSQL cloud instance initialized on April 7 at 14:21.
  • Streamlit Community Deployment: Cloud environment instantiated on April 13 at 22:33, exposing a live interface.
  • Live API Testing: Integration of meteorology feeds directly within the runtime environment.

The Initial Architectural Risk

The system was consolidated into a single monolithic execution file (app.py). This caused issues during real-world field-deployment:

  • Network Latency: Raw database queries compiled without caching filters siphoned data on every click event.
  • NameError Failures: Implicit state dependencies crashed components if frames loaded out of sequence.

v2.x – v3.0: Exploding the Monolith Architecture

To eliminate code redundancy and make the application enterprise-ready, the monolithic codebase was split into clean layers:

src/database.py: Handles clean, Just-In-Time connection pooling.

src/engine/: Isolate pure algorithmic operations from interface state.

views/: Presentation layers focused solely on rendering logic.

controllers/: Validation layers filtering requests before execution.

Persistent State Hydration

Introduced a secure CookieController() framework inside the main application file. This keeps user login tokens securely cached for 30 days, avoiding repetitive logins on field equipment.

User preferences (like localized UI language) automatically load from database user profiles directly into active state bags upon login.

Phase 3: Deep Refactoring

v3.5: Centralized Internationalization (i18n)

Single Source of Truth

All user-facing UI strings were extracted out of page layouts and centralized inside src/lang.py within a core TRANSLATIONS dictionary map.

Eliminated hardcoded strings by introducing a uniform lookup gate: t("key_identity", lang).

Defensive Language Fallbacks

The translation lookup engine is protected against missing strings: if a specific string key is missing from the active language file (e.g., English), it safely defaults to Swedish without interrupting UI rendering pipelines.

This decoupling makes adding new languages simple, requiring zero structural updates to view layout engines.

Phase 3: Deep Refactoring

v3.0 Log Revolution: Thread-Isolated Observability

The Non-Blocking Performance Problem

Writing network event entries (such as coach logins and user activity) can introduce latency and degrade the mobile experience. To solve this, logging was moved to an asynchronous architecture inside logger.py.

UI interactions trigger threading.Thread actions instantly, offloading logging operations to background processes and keeping user interactions lag-free.

The Multi-Thread Context Fix

Streamlit's context wrappers (like st.context.headers) lose connection parameters when shifting between threads. Coach Engine solves this by capturing client metadata before spawning a worker thread:

# Capture metadata on the primary user thread client_ip = headers.get("X-Forwarded-For") thread = threading.Thread(target=log_event_sync, args=(..., client_ip)) thread.start()

Immutable Data Governance: PostgreSQL Audit Triggers

Decoupling Security Fields

Critical audit tracking was moved out of app code and placed directly onto database schemas via native triggers (AFTER UPDATE ON public...).

Even if an attacker bypasses application filters or interacts with tables through external management consoles, database engines capture and preserve the modifications.

Payload Differential Ledgers

Audit functions automatically compute difference snapshots using native JSONB fields, preserving records across standard update sequences:

  • al_old_payload: Tracks the historical database row values.
  • al_new_payload: Stores the updated row values.
  • v_actor_id: Tracks changes to user identities directly using PostgreSQL's auth.uid()::text helper.

Phase 4: Parametric Programming

v4.0: The Brain: Trestegsraket Generation Blueprint

Workout generation was rebuilt to move past random exercise pickers. Sessions are built through an ordered three-stage pipeline:

1. Rules Core (rules.py)

Validates basic limits like max equipment counts, workout durations, and environmental flags.

2. Select Layer (selector.py)

Filters available movements against real-time weather and geography limits.

3. Architect (architect.py)

Builds the workout structure into final loops like Tabata, Triplets, or EMOM blocks.

Weather Intelligence Layer: Live Meteorological Mutation

1. API Ingestion & JIT Backups

The scheduling engine fetches real-time telemetry from external weather APIs based on a location's exact GPS coordinates, retrieving temperature, wind velocity, and standard WMO weather codes.

If an API call fails or times out while out in the park, the engine safely degrades by pulling the latest cached data profile.

2. Meteorological Mapping

The system passes active weather metrics directly into rules.calculate_weather_id() to translate raw WMO codes into internal, actionable environmental IDs.

WMO_HEAVY_RAIN_CODES = [63, 65, 81, 82] EXTREME_WIND_THRESHOLD = 10.0 # m/s

Managing Infrastructure & Equipment Constraints

Geographic Optimization

The engine queries infrastructure flags for the current location, prioritizing natural terrain features over static movements (e.g., swapping high-knees for hill sprints when a hill is available).

Logistical Guardrails

Enforces a strict gear cap: MAX_EQUIPMENT_TYPES_PER_STATION = 1. This prevents setup confusion and keeps equipment transitions smooth during group rotations.

Exclusive Gear Filters

Scans exercise titles for keywords like TRX, Battle Ropes, or Gummiband to flag exclusive equipment, adjusting group pairings to keep item sharing friction-free.
